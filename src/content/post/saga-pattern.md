## 1. 어떤 문제를 푸는가 — 분산 트랜잭션

모놀리식에서는 여러 작업을 한 DB 트랜잭션으로 묶을 수 있다.

```sql
BEGIN;
  INSERT INTO orders ...;
  UPDATE inventory SET stock = stock - 1 ...;
  INSERT INTO payments ...;
COMMIT;
```

ACID가 알아서 원자성을 보장해준다.

마이크로서비스에서는 같은 작업이 이렇게 흩어진다.

- Order Service → 주문 저장
- Payment Service → 결제 처리
- Inventory Service → 재고 차감
- Shipping Service → 배송 예약

**서비스마다 DB가 다르다.** 하나의 트랜잭션으로 묶을 수 없다. 2PC(Two-Phase Commit) 같은 분산 트랜잭션은 가용성/성능 문제로 MSA에서는 거의 쓰지 않는다.

> "결제는 됐는데 재고 차감이 실패하면?" 을 어떻게 처리할 것인가.

이게 Saga가 푸는 문제다.

## 2. Saga의 아이디어

전체를 하나의 큰 트랜잭션으로 묶는 대신,

> 여러 개의 **로컬 트랜잭션**으로 쪼개고, 중간에 실패하면 앞 단계들을 되돌리는 **보상 트랜잭션(Compensating Transaction)** 을 실행한다.

```
T1: 주문 생성    ──→ 실패 시 보상 없음 (시작점)
T2: 결제 승인    ──→ 실패 시 C1 (주문 취소)
T3: 재고 차감    ──→ 실패 시 C2, C1 (결제 취소, 주문 취소)
T4: 배송 예약    ──→ 실패 시 C3, C2, C1
```

각 단계는 자기 서비스의 로컬 트랜잭션이라 ACID로 보장된다. 전체 일관성은 **최종 일관성(eventual consistency)** 으로 수렴한다.

### 여담 — 왜 "Saga"라고 부를까

이름은 1987년 프린스턴의 Hector Garcia-Molina & Kenneth Salem 논문 [Sagas](https://www.cs.cornell.edu/andru/cs711/2002fa/reading/sagas.pdf)에서 유래했다. 원래는 MSA가 아니라 **DB의 Long-Lived Transaction(LLT)** 문제를 풀기 위한 개념이었다.

문학에서 **saga**는 여러 개의 짧은 이야기(챕터)가 이어져 하나의 큰 서사를 이루는 장편 서사시를 뜻한다 (북유럽 신화의 saga가 대표적). 트랜잭션에 비유하면:

- 여러 챕터 → 여러 개의 로컬 트랜잭션
- 각 챕터가 그 자체로 완결 → 각 트랜잭션이 독립적으로 커밋
- 챕터가 모여 긴 서사 → 작은 트랜잭션들이 모여 비즈니스 프로세스 완성

즉 "하나의 거대한 트랜잭션"이 아니라 **"여러 작은 이야기로 이어진 긴 서사"** 라는 비유다. 이 개념을 30년쯤 뒤 Chris Richardson 같은 사람들이 MSA 분산 트랜잭션에 가져오면서 지금의 Saga 패턴이 됐다.

## 3. 두 가지 스타일 — Choreography vs Orchestration

Saga를 구현하는 방식은 크게 두 가지다.

### 3-1. Choreography (이벤트 기반)

각 서비스가 이벤트를 듣고 반응한다. **중앙 조율자가 없다.**

```
Order Service     Payment Service    Inventory Service
     │                  │                    │
     │─OrderCreated────→│                    │
     │                  │─PaymentApproved───→│
     │                  │                    │─StockReserved─→ ...
     │←─StockFailed────────────────────────│
     │  (보상: 결제 취소 이벤트 발행)
```

- **장점**: 결합도 낮음. 새 서비스 추가가 쉬움.
- **단점**: 흐름이 코드에 분산돼서 **전체 그림 파악이 어렵다**. 디버깅, 모니터링 까다로움.
- **적합**: 단계가 적고(3~4개), 서비스 간 의존이 단순할 때.

### 3-2. Orchestration (중앙 코디네이터)

**Orchestrator(코디네이터)** 가 각 서비스에 명령을 내리고 응답을 받아 다음 단계를 결정한다.

```
              ┌─────────────────┐
              │  Saga           │
              │  Orchestrator   │
              └────────┬────────┘
                       │ 명령
        ┌──────────────┼──────────────┐
        ↓              ↓              ↓
   Order Svc     Payment Svc    Inventory Svc
```

- **장점**: 흐름이 한 곳에 명시적으로 있어 **이해/디버깅이 쉽다**. 복잡한 분기 표현 가능.
- **단점**: Orchestrator가 단일 장애점이 될 수 있음. 비즈니스 로직이 코디네이터에 몰림.
- **적합**: 단계가 많고(5개 이상), 분기/조건이 복잡할 때.

> 실무에선 **Orchestration이 더 자주 권장**된다. 흐름이 명시적이고, Temporal·Camunda·AWS Step Functions 같은 워크플로우 엔진이 잘 받쳐준다.

## 4. 보상 트랜잭션 설계 시 주의점

### 4-1. 보상은 의미적 롤백이지 진짜 롤백이 아님

DB 롤백처럼 "없던 일"로 만들 수는 없다. 이미 발생한 사건을 **상쇄하는 새로운 트랜잭션**일 뿐이다.

- 결제 승인의 보상 = 환불 (결제 row를 지우는 게 아님)
- 재고 차감의 보상 = 재고 원복
- 메일 발송의 보상 = "취소 메일" 추가 발송 (이미 보낸 메일은 못 지움)

### 4-2. 모든 액션이 보상 가능한 건 아니다

**비가역(non-compensatable)** 액션이 있다.
- 외부 결제 게이트웨이 호출 (취소 정책에 따라 시간 제약)
- 이메일/SMS 발송
- 실물 배송 시작

이런 단계는 **Saga 후반부로** 배치한다. 앞 단계에서 실패할 가능성을 다 거른 뒤 마지막에 실행하면 보상할 일이 줄어든다. 이를 **Pivot Transaction** 이라 부른다 — 그 이후로는 무조건 성공시켜야 한다.

### 4-3. 보상 트랜잭션도 실패할 수 있다

보상 자체가 실패하면? 재시도가 기본이고, 최종적으로는 **운영자 개입이 필요한 dead letter 큐로** 보낸다. 모든 분산 시스템의 끝은 결국 사람이 본다.

## 5. 격리(Isolation)가 없다는 점

ACID의 I(격리)가 빠진다. Saga 진행 중에 **중간 상태가 다른 트랜잭션에 보일 수 있다.**

예: 주문은 생성됐고 결제는 처리 중인데, 다른 사용자가 그 주문을 조회 → "처리 중" 상태가 노출됨.

대응책:
- **Semantic Lock**: 진행 중인 리소스에 "처리 중" 플래그를 둬서 다른 Saga가 못 만지게 함.
- **Commutative Updates**: 순서가 바뀌어도 결과가 같도록 설계 (예: 잔액 -100 / +50 어떤 순서든 결과 동일).
- **Pessimistic View**: 사용자에게 보수적인 값을 보여줌 (예: 차감 중인 재고는 미리 빼서 보여주기).
- **Re-read Value**: 커밋 시점에 값을 다시 읽어 검증.
- **Version File / By Value**: 동시성 제어를 위한 버전 관리.

## 6. Outbox 패턴과의 관계

Saga의 각 단계는 보통 **이벤트나 메시지로** 다음 서비스에 전달된다. 이때 "DB 저장 + 이벤트 발행"의 원자성이 또 문제가 된다 → **Outbox 패턴**으로 해결.

> Saga = 큰 그림 (서비스 간 트랜잭션 조율)
> Outbox = 각 스텝에서 이벤트를 안전하게 발행하는 메커니즘

둘은 거의 항상 같이 쓰인다. Saga만 도입하고 Outbox 없이 직접 발행하면, **Saga 자체가 dual-write 문제로 깨진다.**

## 7. 자주 하는 실수

- **보상 가능성을 고려 안 한 단계 순서** — 비가역 액션을 앞쪽에 두면 보상이 불가능해진다.
- **Saga 상태를 메모리에만 보관** — 코디네이터가 죽으면 진행 중이던 Saga가 유실. 반드시 영속화.
- **타임아웃 설계 누락** — 단계가 무한 대기에 빠지면 Saga 전체가 멈춤. 모든 단계에 타임아웃 + 재시도 정책.
- **컨슈머 멱등성 미보장** — Outbox와 마찬가지로 메시지가 중복 도착할 수 있다. 멱등 처리 필수.
- **모든 트랜잭션을 Saga로** — 단일 서비스 안에서 끝나는 일에는 과한 도구. 정말 서비스 경계를 넘는 트랜잭션에만.

## 8. 언제 쓰고 언제 안 쓰나

**쓸 때**
- 여러 서비스에 걸친 트랜잭션 일관성이 필요할 때
- 2PC를 쓸 수 없거나 쓰고 싶지 않을 때 (사실상 MSA의 기본)
- 최종 일관성으로 충분한 비즈니스 도메인

**안 써도 될 때**
- 단일 서비스/DB로 끝나는 작업 (그냥 로컬 트랜잭션)
- 강한 일관성이 필수인 도메인 (계좌 이체 등 — 이런 건 보통 한 서비스로 묶는다)

## 정리

- Saga는 **여러 서비스에 걸친 분산 트랜잭션**을 로컬 트랜잭션 + 보상 트랜잭션의 시퀀스로 푸는 패턴이다.
- 두 스타일이 있다: **Choreography**(이벤트 기반, 결합도 ↓) / **Orchestration**(중앙 코디네이터, 가시성 ↑).
- 보상은 의미적 롤백이며, 비가역 액션은 후반부에 배치(Pivot Transaction)한다.
- ACID의 격리(I)가 없으므로 Semantic Lock 등으로 보완한다.
- 각 스텝의 이벤트 발행 신뢰성은 **Outbox 패턴**과 함께 확보한다.
